# 1. Every new issue starts with the `triage` label

All feature requests, ideas, documentation proposals, and integration concepts **must** begin with:

### ‚úî Approve ‚Üí move to a child repository

---

# 3. When an issue becomes a project

A project is created when:

- The feature is large  
- Multiple contributors are expected  
- Multiple repos are involved  
- A roadmap is needed  

Each project must include:

- A **Project Plan** issue  
- A **backlog** of tasks  
- A **delivery order**  
- A **clear goal**  
- A **scope definition**  

---

# 4. When an issue goes directly to a child repository

This applies to:

- Small VS Code extension features  
- Small CI/CD improvements  
- Documentation updates  
- Minor integrations  
- Single-step enhancements  

These do **not** require a project.

---

# 5. Labels used during triage

| Label | Meaning |
|-------|---------|
| `triage` | Waiting for review |
| `approved` | Accepted and ready for next step |
| `needs-info` | More details required |
| `rejected` | Not aligned with vision |
| `project` | Part of a larger project |
| `backlog` | Approved but not scheduled |
| `priority: high` | Important |
| `priority: medium` | Useful |
| `priority: low` | Nice-to-have |

---

# 6. Contributor expectations

Contributors should:

- Provide clear descriptions  
- Follow the issue templates  
- Link related issues  
- Respect the triage process  
- Wait for approval before starting work  

---

# 7. Maintainer expectations

Maintainers should:

- Review triage issues regularly  
- Communicate decisions clearly  
- Create projects when needed  
- Keep backlogs clean  
- Support contributors  

---

# 8. Why this process exists

This structure ensures:

- No chaos  
- No lost ideas  
- Clear priorities  
- Predictable workflows  
- A welcoming contributor experience  
- A scalable ecosystem
